### Points à trancher → décisions
- **Forme du journal** : table dédiée `action_log`, PAS réutilisation de `generation_events`. Motif vérifié dans le code : `generation_events` = observabilité moat (edit_distance/tokens SM-1), écrit uniquement à l'ENVOI et seulement pour une génération → mauvaise forme + mauvais déclencheur pour un journal transverse (createContact/import sans envoi). Une table à colonne `user_id` hérite gratuitement du scoping de la porte (`scoped.ts`).
- **Mapping op→inverse** : contractualisé dans `action-inverse-map.md`. Nuance load-bearing = la dédup d'inc.3 (`create`/`bulkCreate` fusionnent/réactivent) ⇒ ops `merged`/`reactivated` exigent `prevState` ; inverse = restauration, jamais archivage aveugle (CAP-3). `created` ⇒ archivage. Brouillon ⇒ retrait soft.
- **Granularité** : par TOUR (un run `runAgentChat`), pas par message chat ni par tool. Réutilise le fait serveur « ≥1 write-tool » d'inc.2. Rewind = tour ciblé + tours postérieurs, LIFO.
- **Matérialisation du point de rewind sans historique persistant** : `turnId` renvoyé in-band dans le flux UI (canal `toUIMessageStreamResponse` d'inc.2, à côté de `didWrite`) ; popup retient `(tour ↔ turnId)` en-session ; seul `action_log` est durable.
- **UI** : affordance « Annuler ce tour » dans le popup, design-system plein (mauve = action). Contrainte.

### Décisions structurantes (au-delà des points listés)
- **Rewind ≠ tool d'agent** : affordance humaine (server action), hors catalogue agent. Parité « l'humain envoie/annule » + évite footgun auto-annulation. `turnId` clos par closure comme `userId`.
- **Journal atomique avec la mutation** (parité `markSent`+`generation_events`, archi « jamais un log async ratable »).
- **Rewind journalisé, non effaçant** (entrée `op="rewind"`) → journal = actif d'audit SaaS complet.
